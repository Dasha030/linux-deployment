# linux-deployment

# Автоматизація розгортання вебзастосунку Notes Service

Цей репозиторій містить результати розробки, безпечного налаштування та автоматизації розгортання трирівневого вебзастосунку в середовищі ОС Linux.

---

## 1. Варіант індивідуального завдання

Згідно з номером у списку групи (**N = 27**), архітектуру та параметри розгортання визначено за формулами:

*   **Тематика застосунку ($V_3 = (27 \bmod 3) + 1 = 0 + 1 = 1$):** **Notes Service** (Сервіс текстових нотаток).
*   **Спосіб конфігурації ($V_2 = (27 \bmod 2) + 1 = 1 + 1 = 2$):** Через конфігураційний файл JSON за фіксованим шляхом `/etc/mywebapp/config.json`.
*   **База даних ($V_2 = (27 \bmod 2) + 1 = 1 + 1 = 2$):** **PostgreSQL**. Налаштовано локальний доступ (`127.0.0.1`) із забороною зовнішніх мережевих підключень.
*   **Порт застосунку ($V_5 = (27 \bmod 5) + 1 = 2 + 1 = 3$):** **3000** (вебсервер Node.js слухає виключно локальний інтерфейс `127.0.0.1:3000`).

## 2. Документація по розробленому веб-застосунку

### Призначення застосунку
Вебзастосунок є сервісом текстових нотаток (Notes Service). Він дозволяє користувачам створювати нотатки, переглядати список усіх збережених записів, а також читати повний текст окремої нотатки за її унікальним ідентифікатором (ID). Сервіс підтримує динамічне визначення типу контенту (Content Negotiation) на основі HTTP-заголовків.

### Як налаштувати середовище для розробки/тестування/запуску
Для локальної розробки та тестування на робочій машині необхідно:
1.  Встановити **Node.js v20 LTS** та менеджер пакетів **npm v10+**.
2.  Встановити та запустити сервер СУБД **PostgreSQL v14+**.
3.  Створити базу даних `mywebapp_db` та користувача `webapp_user`.
4.  Встановити залежності проєкту, запустивши в папці з файлом `package.json` команду:
    pm install
5.  Створити локальний файл конфігурації відповідно до шаблону `config.example.json`.

### Як запустити розроблений веб-застосунок
Перед першим запуском необхідно виконати ініціалізацію таблиць у базі даних (міграцію):
    
    npm run migrate
    
Після успішного створення таблиць, запуск застосунку в режимі сервера виконується командою:
    
    npm start

### Документація по API-ендпоінтах
Службові ендпоінти (Health Checks)
* GET /health/alive — Перевіряє, чи запущений процес Node.js. Повертає текстову відповідь OK з кодом 200.
* GET /health/ready — Динамічно перевіряє наявність активного підключення до бази даних PostgreSQL. У разі успіху повертає OK (200), у разі збою — опис помилки з кодом 500.

Ендпоінти бізнес-логіки
* GET / — Головна сторінка. Повертає перелік усіх доступних маршрутів системи (завжди у форматі text/html).
* GET /notes — Отримання списку нотаток (повертає поля id та title).
* POST /notes — Створення нової нотатки. Приймає JSON-тіло запиту з обов'язковими полями title та content. Повертає об'єкт створеної нотатки з кодом 201.
* GET /notes/<id> — Отримання повного тексту та метаданих конкретної нотатки за її id. Якщо нотатки немає — повертає код 404.

Специфікація форматів відповіді (Обробка заголовка Accept)
Для ендпоінтів /notes та /notes/<id> реалізовано логіку фільтрації контенту:

JSON (За замовчуванням): Якщо заголовок Accept відсутній або містить application/json, дані повертаються як структурований JSON-об'єкт.

HTML: Якщо клієнт передає заголовок Accept: text/html, сервер динамічно генерує та повертає чисту HTML-сторінку.

## 3. Документація по розгортанню (Deployment Guide)
Базовий образ віртуальної машини

Дистрибутив: Офіційний серверний образ Ubuntu 24.04 LTS.Завантажити можна на офіційному репозиторії Canonical (releases.ubuntu.com). Завантажувати файл з розширенням .iso (Ubuntu Server).

Вимоги до ресурсів віртуальної машини

CPU: Мінімум 1 vCPU (рекомендовано 2 vCPU для стабільної компіляції/встановлення пакетів).

RAM: Мінімум 1 ГБ (рекомендовано 2 ГБ, оскільки PostgreSQL та процеси npm під час інсталяції можуть споживати значний обсяг пам'яті).

Disk (HDD/SSD): Мінімум 10 ГБ вільного простору.

Мережа: 1 віртуальний мережевий адаптер у режимі "NAT" (або "Bridge") з доступом до мережі Інтернет для завантаження пакетів.

Спеціальні налаштування при встановленні OS
Під час інсталяції Ubuntu Server рекомендується використовувати стандартну розбивку дисків через менеджер логічних томів LVM (за замовчуванням), залишаючи весь простір під кореневий розділ /.

Обов'язково позначити пункт "Install OpenSSH Server" на етапі встановлення для забезпечення віддаленого доступу.

Вхід на віртуальну машину (Credentials)

Спосіб входу: Через SSH-клієнт або системну консоль гіпервізора (VirtualBox/VMware).

Команда підключення: ssh student@<IP_адреса_ВМ>

Користувач за замовчуванням: student.


Як запустити автоматизацію розгортання

Розгортання системи повністю автоматизовано. Для встановлення середовища "під ключ" потрібно виконати наступні кроки на чистій ВМ:

1 Увійдіть на віртуальну машину під користувачем student.
2 Завантажте репозиторій із кодом:

git clone [https://github.com/your-username/your-repo-name.git](https://github.com/your-username/your-repo-name.git)

    cd your-repo-name 
    
3 Надайте права на виконання скрипту розгортання:

    chmod +x deploy.sh
    
4 Запустіть скрипт (деякі етапи конфігурації системи потребуватимуть введення пароля для `sudo`):

    ./deploy.sh

Скрипт автоматично встановить `curl`, `PostgreSQL`, `Nginx`, налаштує безпеку бази даних, додасть системних користувачів, змонтує Systemd-сервіс `mywebapp.service` та конфігурацію проксі-сервера Nginx.

## 4. Інструкція з тестування розгорнутої системи

Після завершення розгортання, коректність роботи всіх компонентів інфраструктури перевіряється наступним чином:

### 1. Перевірка системних служб

Переконайтеся, що база даних, проксі-сервер та вебзастосунок працюють у фоновому режимі й активні (`active (running)`):

    sudo systemctl status postgresql
    sudo systemctl status nginx
    sudo systemctl status mywebapp.service

### 2. Перевірка ізоляції та обмеження доступу (Nginx)

Службові ендпоінти здоров'я системи мають бути закриті від зовнішніх користувачів на рівні Nginx. Перевіримо це за допомогою HTTP-запитів до вебсервера (порт 80):

    curl -I http://localhost:80/health/alive
    
Очікуваний результат: HTTP-код 403 Forbidden.

Перевірка доступності цих же ендпоінтів безпосередньо зсередини (локальний порт 3000):

    curl -i [http://127.0.0.1:3000/health/ready](http://127.0.0.1:3000/health/ready)

Очікуваний результат: HTTP-код 200 OK та текст OK.

### 3. Тестування бізнес-логіки та Content Negotiation

Перевіримо створення нотатки через API, надіславши POST-запит із JSON-тілом:

    curl -X POST http://localhost:80/notes \
    -H "Content-Type: application/json" \
    -d '{"title": "Тестова нотатка", "content": "Текст лабораторної роботи №1"}'
    
Очікуваний результат: Код 201 Created та JSON-відповідь із присвоєним ID нотатки.

Перевірка отримання даних у форматі JSON:

    curl -H "Accept: application/json" http://localhost:80/notes
    
Очікуваний результат: Структурований рядок JSON зі списком нотаток.

Перевірка отримання даних у форматі HTML (імітація запиту від браузера):

    curl -H "Accept: text/html" http://localhost:80/notes
    
Очікуваний результат: HTML-код сторінки, що містить тег <table border="1"> з існуючими записами.

### 4. Перевірка прав та користувачів Linux

1 Перевірка створення файлу gradebook користувача student:

    ls -l ~/gradebook
    # Має відобразити права доступу -rw------- (доступ лише власнику)
    cat ~/gradebook
    # Має вивести номер варіанту: 27

2 Перевірка обмежених прав користувача `operator`:

    # Перехід під користувача operator (пароль 123456789)
    su - operator
    # Спроба перезапустити наш вебсервер без введення пароля sudo
    sudo systemctl restart mywebapp.service
    # Команда має виконатися успішно без запиту пароля адміністратора
    # Спроба виконати несанкціоновану команду (наприклад, читання логів або встановлення пакунків)
    sudo apt update
    # Система має заблокувати команду з повідомленням про відсутність прав у файлі sudoers
